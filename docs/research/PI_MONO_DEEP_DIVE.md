## System Overview

pi (`pi-mono`) is a lean, hackable coding agent built as a TypeScript monorepo. Its design
philosophy is the opposite of Claude Code's: a **small, layered set of npm packages** where the
LLM abstraction, the agent framework, the coding agent, and the TUI are each independently
publishable. It is the spiritual ancestor of both swe-term (which ports its architecture to
Go/Rust) and Flue (which builds a headless framework on top of `pi-agent-core` + `pi-ai`).

**Tech Stack:**

| Layer | Technology |
|-------|-----------|
| Runtime | Node.js |
| Language | TypeScript |
| Package mgmt | npm workspaces (monorepo) |
| Schema | TypeBox (`Type.Object(...)`) — JSON-Schema-native, Zod-like |
| Extension loading | `jiti` (run TS without a build step) |
| TUI | custom (`@mariozechner/pi-tui`, differential renderer) |
| Session format | JSONL (append-only, tree via `id`/`parentId`) |
| Cancellation | `AbortController` / `AbortSignal` |

**Scale:** the headline number is the `AgentSession` class at **3,060 lines**
(`packages/coding-agent/src/core/agent-session.ts`). The rest of the system is comparatively
small — the value is in the *layering*, not the volume.

## Monorepo Package Layout

Seven packages live under `packages/*/` (all `@mariozechner/*` in this checkout, version
`0.65.2`):

| Package | Role |
|---------|------|
| **`@mariozechner/pi-ai`** | Unified LLM API across ~10 provider families / 17 provider files (OpenAI completions+responses, Anthropic, Google + Vertex + Gemini-CLI, Bedrock, Mistral, Azure, …). Model registry (generated), auth, streaming, cost/usage. |
| **`@mariozechner/pi-agent-core`** | General-purpose agent framework: the `Agent` class (`agent.ts`, ~540 ln), the agent loop (`agent-loop.ts`, ~632 ln: `agentLoop()` / `agentLoopContinue()` / `runAgentLoop()` / `runAgentLoopContinue()`), tool-call orchestration, state, steering/follow-up queues. |
| **`@mariozechner/pi-coding-agent`** | The `pi` CLI: `AgentSession`, built-in tools, extensions, session management, the interactive / print / **rpc** modes. |
| **`@mariozechner/pi-tui`** | Terminal UI library with differential rendering (~10,300 ln across 25 files). |
| **`@mariozechner/pi-web-ui`** | Browser (Lit/mini-lit) components for an AI chat UI. |
| **`@mariozechner/pi-mom`** | Slack bot that delegates to the `pi` coding agent (own `log.jsonl` per channel). |
| **`@mariozechner/pi`** (`packages/pods`) | CLI for running vLLM on GPU pods (`pi-pods` bin) — unrelated to the agent loop. |

The dependency direction is clean: `pi-coding-agent` → `pi-agent-core` → `pi-ai`, with `pi-tui`
as a presentation dependency of the coding agent. (`mom` and `pods` are peripheral consumers,
not part of the core stack.) This layering is the single most important thing pi-mono gets
right, and the reason it is portable at all.

## pi-ai: The Provider Abstraction

`pi-ai` is the bottom layer: one API over many LLM providers. It abstracts authentication
(`AuthStorage`), message formatting, streaming, and cost calculation. A `MODELS` registry
(generated by `scripts/generate-models.ts`) lists models and their configuration.

### Capability compat

Provider quirks are handled through a `compat` field on the `Model` type rather than `if
provider === 'x'` branching everywhere. For the `openai-completions` API, `OpenAICompletionsCompat`
declares **14** capability flags (`packages/ai/src/types.ts:255–283`):

`supportsStore`, `supportsDeveloperRole`, `supportsReasoningEffort`, `reasoningEffortMap`,
`supportsUsageInStreaming`, `maxTokensField`, `requiresToolResultName`,
`requiresAssistantAfterToolResult`, `requiresThinkingAsText`, `thinkingFormat`,
`openRouterRouting`, `vercelGatewayRouting`, `zaiToolStream`, `supportsStrictMode`.

When `compat` is **not** set, `pi-ai` *falls back to URL/provider-substring detection*
(`detectCompat()` in `providers/openai-completions.ts:802+`) — matching the **provider slug or
base URL** against strings like `api.z.ai` (zai), `cerebras.ai`, `deepseek.com`, `openrouter.ai`,
`chutes.ai`, `groq.com`, and `ai-gateway.vercel.sh` (Vercel gateway). So there are two mechanisms
in play: explicit capability flags (the intended path) and URL-substring sniffing (the fallback).

### Usage accounting

`pi-ai` tracks real token usage, not a heuristic. The `Usage` type carries `input`, `output`,
`cacheRead`, `cacheWrite`, and `totalTokens` (for OpenAI, `input` excludes cached tokens and
`totalTokens = input + output + cacheRead + cacheWrite`). This is provider-reported usage for
*accounting*; note this is distinct from *pre-flight* token estimation used to decide when to
compact (see [Compaction](#context-compaction)).

### Streaming

`streamSimple` is the default streaming function. The `Agent` class accepts a `streamFn` option so
callers can substitute a custom streaming implementation (used for testing and for wrapping/
instrumentation).

## pi-agent-core: The Agent Framework

This package contains the **provider-agnostic agent**: an `Agent` class for state management and
tool calling, and the loop functions `agentLoop()` / `agentLoopContinue()` (also surfaced as
`runAgentLoop` / `runAgentLoopContinue`). It deliberately knows nothing about the *coding* domain
— no file tools, no bash — so it can be reused (Flue builds on exactly this layer).

`ThinkingLevel` (reasoning-effort control) lives here and is re-exported by downstream consumers.

---

## The Agent Loop

The loop lives in `packages/agent/src/agent-loop.ts` (~632 lines) and is driven by the `Agent`
class in `agent.ts` (~540 lines). It is the standard agentic cycle, factored into the core
package. The real structure is an **outer follow-up loop wrapping an inner tool+steering loop**:

```
runAgentLoop(agent, input):
  outer loop:                              # follow-up messages
    inner loop while toolCalls or pending: # tool execution + steering
      inject steering messages
      stream assistant response (streamFn, default streamSimple from pi-ai)
      execute tools (via ExtensionRunner; extensions can intercept/block/modify)
      append tool results to history
      pending = getSteeringMessages()
    followUp = getFollowUpMessages()
    if followUp: continue outer loop
    else: break
```

`agentLoopContinue()` resumes a loop from existing state (e.g. after a branch switch or a
follow-up). The loop emits events that the TUI/CLI subscribe to. The `Agent` exposes a `streamFn`
option defaulting to `streamSimple` (`agent.ts`: `this.streamFn = options.streamFn ?? streamSimple`).

## AgentSession: The Orchestrator

**Path:** `packages/coding-agent/src/core/agent-session.ts` (**3,060 lines**)

`AgentSession` is the central object of the coding agent — and the one swe-term explicitly singles
out as a **god object to decompose** (PLAN gap #1). It wraps the `pi-agent-core` `Agent` and
coordinates between the TUI/CLI and the LLM. Its responsibilities (all in one class):

- Agent lifecycle and state access
- Event subscription and **automatic session persistence**
- **Model and thinking-level management** (including `_scopedModels`)
- **Manual and automatic compaction** (with separate `AbortController`s for each)
- **Bash execution**
- **Session switching and branching** (tree navigation)
- **Extension integration** — driving `ExtensionRunner` for tool-call interception
- Retry handling

The class holds a fleet of `AbortController`s — `_compactionAbortController`,
`_autoCompactionAbortController`, `_branchSummaryAbortController`, `_retryAbortController`,
`_bashAbortController` — one per concurrently-cancellable operation. This is the `AbortSignal`
cancellation pattern (PLAN gap #7) applied pervasively.

It also uses `any` in places (`_scopedModels: Array<{ model: Model<any>; … }>`, `Model<any>`),
despite the project's TypeScript-strict aspirations (PLAN gap #12).

## Tool System

Tools are plain TypeScript objects with:

- `name`
- `label`
- `description`
- `parameters` — defined with **TypeBox** (`Type.Object({...})`), which is JSON-Schema-native and
  serves the same role as Zod or a hand-written JSON Schema.

Tool *execution* is driven through the `AgentSession`'s event loop and the `ExtensionRunner`,
which gives extensions a hook point to intercept, block, or rewrite a tool call before/after it
runs.

Built-in coding tools include the usual file operations (read, write, edit) and shell execution
via a `bash` tool.

---

## Extension System

**Mechanism:** extensions are TypeScript modules that `export default` a factory function. The
factory receives an `ExtensionAPI` object and can call `pi.registerTool()` (among other hooks) to
contribute tools, intercept tool calls, and inject context.

**Loading:** extensions are loaded with **`jiti`**, which executes TypeScript directly with no
prior compile step. A tool registered by an extension is **immediately available to the LLM
without a reload** — a genuinely nice DX property for live iteration.

This is also PLAN gap #2: the extension system depends on JS/TS **dynamic `import()`** at runtime,
which is natural in Node but is exactly the dynamism swe-term wants to avoid (it chooses Starlark
tool defs + Go interface hooks instead).

## Session Storage (JSONL Tree)

Sessions persist as **append-only JSONL**, one file per session named
`{timestamp}_{sessionId}.jsonl` (not a single `log.jsonl` — that name belongs to the `mom` Slack
bot). Each entry has an `id` and a `parentId`, so the file encodes a **conversation tree** that
supports in-place branching. The `SessionManager` owns persistence with operations like
`_appendEntry()` (private) / `appendMessage` / `appendCompaction` / `appendCustomEntry`,
`buildSessionContext()`, and branch path resolution.

There is **no on-disk index**, but `SessionManager` does build an **in-memory `byId: Map`**
(`_buildIndex()`) on load; branch/path resolution then walks the `parentId` chain through that
map. So loading a session is still O(n) file read + index build, and deep navigation is
tree-walking (PLAN gap #4, which swe-term fixes with a real SQLite store). Session files live under
`{getAgentDir()}/sessions/<encoded-cwd>/`; the agent dir is overridable via the
**`PI_CODING_AGENT_DIR`** env var.

## Context Compaction

When the conversation exceeds a configurable token threshold, pi automatically **summarizes older
content** to reclaim context while preserving recent turns. The flow:

1. Find a **cut point** in the message history.
2. Extract the messages to summarize.
3. Generate a summary via the LLM.
4. Append a `CompactionEntry` to the session log.

Compaction can also be triggered manually with `/compact`. `AgentSession` owns both the manual and
automatic compaction paths (each with its own `AbortController`).

> **Note on token counting:** `pi-ai` reports *real* usage from providers for accounting, but the
> swe-term gap analysis (PLAN gap #5) flags pi-mono's *pre-flight* estimation as a `chars/4`
> heuristic — i.e. the decision of *when* to compact uses an approximation rather than a true
> tokenizer. swe-term replaces this with `tiktoken-go` or a Rust tokenizer subprocess.

## pi-tui: The Terminal UI

`@mariozechner/pi-tui` is a **custom terminal UI library** built around **differential
rendering** — it computes and redraws only the changed regions of the screen, reducing flicker.
The `TUI` class (which `extends Container`) is the rendering core, tracking `previousLines` for the
diff; exported components include `Box`, `Text`, `TruncatedText`, `Markdown`, `Editor`, `Input`,
`Loader`, `CancellableLoader`, `SelectList`, `SettingsList`, `Spacer`, `Image`, and
`ProcessTerminal`.

The coding agent's `InteractiveMode`
(`packages/coding-agent/src/modes/interactive/interactive-mode.ts`, ~4,690 lines) handles rendering
and user interaction, delegating all business logic to `AgentSession`. The renderer moved off
`chalk` toward a dedicated theme system.

This is a hand-rolled renderer — the `pi-tui` package source is **~10,300 lines across 25 files**
(e.g. `editor.ts` ~2,231 ln, `keys.ts` ~1,357 ln, `tui.ts` ~1,241 ln) — which swe-term replaces
with `ratatui` (Rust) or Bubble Tea (Go) rather than maintaining a bespoke diff renderer.

## Modes (Interactive / Print / RPC)

The coding agent separates **three modes** from the session (`packages/coding-agent/src/modes/`,
routed in `main.ts`):

- **Interactive mode** (`modes/interactive/interactive-mode.ts`, ~4,690 ln) — the full TUI
  experience, delegating to `AgentSession`.
- **Print mode** (`modes/print-mode.ts`) — non-interactive output (stdout), for scripting/piping.
- **RPC mode** (`modes/rpc/rpc-mode.ts`, ~695 ln) — a JSON stdin/stdout protocol (`runRpcMode()` +
  `RpcClient`) for programmatic embedding. This is a first-class source mode, not a dist-only
  artifact.

All three drive the same `AgentSession`, so business logic is shared and the UI is swappable — the
same "frontends are interchangeable" idea Codex enforces via its protocol seam, here achieved via a
shared orchestrator object.

> **No built-in MCP.** Unlike Codex/Claude Code, pi-coding-agent ships **no built-in MCP client**;
> the README directs users to write an extension or use CLI skills instead. (swe-term treats MCP as
> a first-class `Tool` source.)

## Key Design Patterns

### 1. Layered, independently-publishable packages

`pi-ai` (providers) → `pi-agent-core` (agent) → `pi-coding-agent` (coding) + `pi-tui` (UI). Each is
usable on its own; Flue consumes the lower two directly. This is the cleanest layering of any agent
surveyed and the reason pi-mono is *portable*.

### 2. Provider-agnostic agent core

`pi-agent-core` knows nothing about coding. The coding tools live one layer up. This is the
separation Claude Code never made.

### 3. Capability flags over provider branching (mostly)

`Model.compat` declares what a model supports, instead of scattering `if provider === …`. The
fallback to URL detection is the weak spot.

### 4. Zero-build extensions via `jiti`

Drop a `.ts` file, it's loaded and its tools are live immediately. Excellent iteration speed; deep
Node coupling.

### 5. JSONL tree for branchable history

Append-only log with `id`/`parentId` gives free branching and a human-readable session file — at
the cost of an O(n) file read + in-memory `byId` index build on load, and `parentId` tree-walking
for branch/path resolution (no on-disk index).

### 6. `AbortController` everywhere for cancellation

Every long-running operation (compaction, retry, bash, branch summary) gets its own signal. Maps
directly to Go's `context.Context` in the port.

### 7. Orchestrator-as-god-object

`AgentSession` centralizes everything. Great for discoverability, poor for testability — and the
explicit thing swe-term decomposes.

## Why swe-term Ports It

`../core/PLAN.md` states the thesis directly: *"Porting the architecture of pi-mono — without its
gaps."* pi-mono is chosen as the reference because its **layering is right** (provider / agent /
coding / UI) and its **scope is sane** (a hackable agent, not a 512K-LoC product). swe-term keeps
the layering and the loop, and fixes the twelve enumerated gaps — chiefly: decompose the
`AgentSession` god object, replace dynamic-`import()` extensions with Starlark + Go interfaces,
replace URL-substring provider detection with a clean `Provider` interface, replace JSONL
tree-walking with SQLite, and replace `chars/4` estimation with a real tokenizer.

See `../core/PLAN.md` → "pi-mono Gap Analysis" for the full 12-row table; the critique companion to
this document (`PI_MONO_CRITIQUE.md`) expands each gap against the Harness philosophy.
